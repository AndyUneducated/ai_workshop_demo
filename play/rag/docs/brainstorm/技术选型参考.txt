技术选型参考

一、向量数据库对比

| 方案 | 优势 | 劣势 | 适合场景 |
| ChromaDB | 轻量、Python 原生、零运维 | 不支持分布式 | 中小规模（<10万文档） |
| Milvus | 高性能、支持分布式 | 运维复杂、资源消耗大 | 大规模（>100万文档） |
| pgvector | 复用现有 PostgreSQL | 性能一般、功能有限 | 想最小化基础设施变更 |

结论：公司 4327 篇文档属于中小规模，ChromaDB 或 pgvector 均可满足。考虑到团队已有 PostgreSQL 经验，pgvector 的运维成本最低。

二、Embedding 模型对比

| 模型 | 维度 | 中文效果 | 部署方式 |
| text-embedding-3-small | 1536 | 优秀 | OpenAI API |
| nomic-embed-text | 768 | 良好 | 本地 Ollama |
| bge-large-zh-v1.5 | 1024 | 最佳中文 | 本地 HuggingFace |

结论：如果优先中文效果且希望本地部署，推荐 bge-large-zh-v1.5；如果接受 API 调用，text-embedding-3-small 综合最优。

三、LLM 对比

| 模型 | 优势 | 成本 |
| GPT-4o | 回答质量最高 | 约 $0.01/次查询 |
| GPT-4o-mini | 性价比最高 | 约 $0.001/次查询 |
| Qwen2.5-32B（本地） | 零 API 成本、数据不出网 | 需要 A10 GPU |

结论：MVP 阶段建议 GPT-4o-mini（快速验证），上线后可切换为本地 Qwen2.5 降低长期成本。

四、前端方案

- 方案 A：嵌入现有 OA 系统（钉钉/飞书 H5 应用）——用户无需切换入口
- 方案 B：独立 Web 应用（React）——开发更灵活，但用户需记住新地址
- 方案 C：企业微信/钉钉机器人——最低开发成本，但交互能力有限

结论：首期推荐方案 B（团队熟悉 React，两天可出界面），二期再考虑嵌入 OA。
