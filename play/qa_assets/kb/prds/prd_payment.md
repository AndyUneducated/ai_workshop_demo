# 支付 / 结算模块 PRD (历史 v2)

## 背景

订单完成后调用支付网关 (Stripe / 微信 / 支付宝), 处理回调并 update order status.

## 关键流程

1. 用户提交支付 → 创建 payment_intent, 状态 = pending
2. 重定向到第三方支付页 (或弹出 SDK)
3. 第三方异步回调 → 验签 → 更新 payment_intent 状态 → 触发订单状态机
4. 失败 / 超时 → 自动重试 3 次 (指数退避), 仍失败标记 manual_review

## 关键约束

- 回调接口必须验签 (HMAC-SHA256), 防伪造回调
- 同一 order_id 重复回调要幂等 (已处理过的直接返回 200)
- 超时阈值: 3 分钟未收到回调视为失败, 起对账任务
- 金额比对: 第三方回调里的金额必须 = 订单金额, 不一致直接拒绝

## 教训

1. **重放攻击**: 黑客抓包重发支付成功回调, 账户余额异常增加. 修复: 回调
   payload 里加 nonce + timestamp, 5 分钟外的拒绝, 同 nonce 一次性入 Redis.
2. **金额单位**: 微信回调返回"分", Stripe 返回"美分", 我们 DB 存"元". 一次因为
   忘记转换, 用户付 1 块钱被记成 100 块. 修复: 在网关 SDK 层统一换算到"元 + decimal", 业务层永远拿元.
3. **退款竞争条件**: 用户在退款流程中再次发起支付, 状态机被打乱. 修复: order_id
   级别加分布式锁, 整个支付 / 退款流程都在锁内.

## 测试覆盖要点

- 正常支付 / 支付失败 / 用户取消
- 回调验签失败 / payload 篡改 / 重放
- 金额不匹配
- 多种支付方式 + 多币种
- 退款 / 部分退款
- 网络超时 + 重试
